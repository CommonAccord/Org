CommonAccord – System description

 

General description

 

The template system allows us to generate easily contract proposals, NDAs, MOUs, etc…

 

In all cases, a typical legal document is composed of at least 2 parts:

-           Doc_Top is the file where the specific parameters for a deal are defined. For example name of parties, effective date, etc…

-           Doc_Core is the actual generic document codified so that parameters that may vary are replaced by variables that will be defined in the Top file.

 

Syntax

 

All documents are a list of value-pairs that are assembled by the CommonAccord software to produce the final document.

The system will assign values to parameters as specified in each document, starting from the initial document being rendered, and then looking into other files/documents mentioned in this initial document.

Once a value has been assigned to a parameter it is not replaced.

In the end if no value has been assigned to a parameter, it is listed at the bottom of the document when it is displayed.

 

For example:

-           If I start from BB_NDA_Top and Content=MutualNDA then this is the value that will be used.

-           But if the value Content is not assigned in BB_NDA_Top, and BB_NDA_Top refers to document BB_NDA_Core, then the system will go and look into BB_NDA_Core to assign the value.

 

Documents typically include the following:

1.       Reference to other files with content that is included in the documents. For example:

=[BeBound/BB_Core]

2.       Values for specific parameters. For example:

Company={P1.Nom} or Cust.Ref=OPERATOR

3.       Content=list of parameters that constitute the core of the document. For example:

Content={MOU} or Content=<p>{Distrib}</p><p>{Scope}</p><p>{Commission}</p><p>{Prospects}</p>

 

Parameters within files:

-           It is possible to specify parameters from another file. It looks like this:

P1.=[BeBound/BB_ID_XXX]

In this case, the file BB_ID_XXX can include parameters such as Name or Address, and then it is possible to refer to any parameter from this file with the “P1.” prefix, for example: P1.Name or P1.Address.

Types of files

 

For Be-Bound we have established the following list of files:

 

1.       BB_ID:

Details for Companies or People. The file includes the following types:

=[BeBound/BB_Type_Individu] if this is for an individual

=[BeBound/BB_Type_Societe_FR] for French companies

=[BeBound/BB_Type_Societe_US] for US companies

=[BeBound/BB_Type_Societe] for companies in other countries

 

2.       BB_Contract:

BB_Contract includes several parts:

-           BB_Contract_Core is the document itself for V2 contracts

-           BB_ContractV3_Core is the document for V3 contracts (no mention of BMA there)

-           BB_Contract_Top is a sample of what can be done for a given deal. For actual deals Top is replaced by the name of the company.

-           BB_Contract_Services is the standard SOW. This can be manually replaced with the actual SOW sent to Customers as needed. In this case, the parameter Contract.Services is replaced with a placeholder

-           BB_Contract_Revenue is the exhibit that defines the revenue sharing schema

-           BB_Contract_SLA is the exhibit that defines the SLA

-           BB_Contact_Other is the exhibit that defines additional responsibilities for the parties as needed

 

3.       BB_TermofUse

BB_TermofUse includes the following parts:

-           BB_TermofUse_Core is the document itself

-           BB_TermofUse_Top is a sample of what can be done for a given deal. For actual deals Top is replaced by the name of the company.

 

4.       BB_Distrib

-           BB_Distrib_Core is the document itself

-           BB_Distrib_Top is a sample of what can be done for a given deal. For actual deals Top is replaced by the name of the company.

-           BB_Distrib_Scope is the Exhibit A that defines the scope for the distributor/agent

-           BB_Distrib_Commission is the Exhibit B that defines the commission for the distributor/agent

-           BB_Distrib_Prospects is the Exhibit C that defines the list of already identified prospects that the distributor or agent is in contact with

 

5.       BB_MOU

-           BB_MOU_Core is the document itself

-           BB_MOU_Top is a sample of what can be done for a given deal. For actual deals Top is replaced by the name of the company.

-           BB_MOU_Scope is the Exhibit A that defines the Scope of the MOU

 

6.       BB_NDA

-           BB_NDA_Core is the list of NDA clauses for both Mutual and Non-Mutual NDAs

-           BB_NDA_Top is a sample of what can be done for a given deal. For actual deals Top is replaced by the name of the company. The file specifies whether the NDA should be MutualNDA or NonMutualNDA depending on what is needed. Text for both is in the BB_NDA_Core document.

 

7.       Misc. Files

 

7.1.             BB_Apps: list of applications available. This is used in the SOW that is included in the contrat. The doc includes SOW stuff as well actually.

 

7.2.             BB_Blocks: signature blocks that are used in various documents

 

 

7.3.             BB_Core: File to be called in all other files that includes the [Model.Root]. This also includes the Company Name and Address for the Footer in the documents

 

7.4.             BB_Definitions: list of definitions to add at the top of the contract. Some definitions can be re-used in other docs as well as needed.

 

7.5.             BB_GeneralConditions: standard clauses that can be used across several documents, for example “No Waiver” or “Entire Agreement” clauses

 

7.6.             BB_Locales: various variations for contacts. For exemple variations related to the business model (subscriptions vs. pack) or related to the geography (French clauses vs. US clauses)

 

7.7.             BB_Type: formatting for various BB_ID files, so that the data is presented in a readable format and can be used easily

 

A few Key Parameters

 

1.       In general for most contracts, the Top part includes a reference to 2 parties, an effective date and Governing Law:

 

-           P1.=[BeBound/BB_ID_XXX] for Party 1 – typically Be-Bound or Be-Bound Inc

-           P2.=[BeBound/BB_ID_YYY] for Party 2

-           EffectiveDate= 

-           GoverningLaw.Country=

-           GoverningLaw.City=
